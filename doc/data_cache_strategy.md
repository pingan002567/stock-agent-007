# 数据分层与缓存策略

> 项目: stock-agent-001 | 日期: 2026-06-07

---

## 1. 数据源限流约束

| 数据源 | 限流机制 | 保守间隔 |
|--------|---------|---------|
| akshare | 无显式限制，底层爬取东方财富/新浪，高频触发反爬 | ≥3s/次 |
| yfinance | 已内置 `_min_call_interval = 2.0s` | ≥2s/次 |
| tushare | Token 级别限流（免费版 ~200次/分钟） | ≥0.3s/次 |

**核心原则**：SQLite 是唯一可靠缓存层，内存缓存只是 SQLite 的"热读窗口"。

---

## 2. 数据六层分类

```
L0 静态 ─ 股票主表
L1 低频 ─ 财报数据
L2 日级 ─ K线历史
L3 分钟级 ─ 板块/市场综览
L4 实时 ─ 实时行情
L5 即时 ─ 情报/新闻
```

### 2.1 L0 股票主表

| 属性 | 值 |
|------|-----|
| 数据量 | 5000+ 条 A 股 |
| 更新频率 | 季度（新股上市/退市） |
| 主存储 | SQLite |
| 更新策略 | 首次空表时同步下载（~35s），后续每周日凌晨 3:00 后台静默更新 |
| API 调用 | 1 次/周 |

### 2.2 L1 财报数据

| 属性 | 值 |
|------|-----|
| 数据量 | ~5 条/股（最近 5 期） |
| 更新频率 | 季度 |
| 主存储 | SQLite |
| 内存缓存 TTL | 3600s |
| 更新策略 | 按需拉取，SQLite 缓存 7 天 |
| API 调用 | 首次 1 次/股，7 天内 0 次 |

### 2.3 L2 K 线历史

| 属性 | 值 |
|------|-----|
| 数据量 | ~250 条/股/年（日线） |
| 更新频率 | 每日收盘后 |
| 主存储 | SQLite |
| 内存缓存 TTL | ∞（依赖 SQLite 缓存，不设过期） |
| 更新策略 | 首次请求拉 1 年数据写 SQLite，后续只拉最近 N 日增量 |
| 缓存命中判断 | SQLite 最新日期 ≤ 1 个交易日 && 条数 ≥ 请求天数 |
| API 调用 | 每只股票 1 次/年（后续只拉增量 1 日） |

```
用户请求 AAPL K线(30天):
  1. 查 SQLite stock_daily: 最新日期? 条数 ≥ 30?
     ├── YES → 直接返回（0 API 调用）
     └── NO → 需要拉取
        ├── 有缓存但数据不够 → 计算缺少天数，只拉增量
        └── 无缓存 → 首次拉 1 年数据，写 SQLite
  2. 内存缓存不设 TTL
```

### 2.4 L3 板块/市场综览

| 属性 | 值 |
|------|-----|
| 数据量 | ~30 条/次 |
| 更新频率 | 盘中分钟级 |
| 主存储 | 内存 |
| 内存缓存 TTL | 300s（5 分钟） |
| 更新策略 | 按需拉取 |
| API 调用 | 盘中 ~12 次/小时 |

### 2.5 L4 实时行情

| 属性 | 值 |
|------|-----|
| 数据量 | 1 条/股 |
| 更新频率 | 盘中 60s，盘后不更新 |
| 主存储 | SQLite |
| 内存缓存 TTL | 盘中 60s，盘后 3600s |

```
用户请求 AAPL 行情:
  1. 查 SQLite stock_quote: created_at 距现在 < 60s?
     ├── YES → 直接返回
     └── NO → 拉取
        ├── 盘中 (9:30-15:00 工作日) → 拉取，写 SQLite
        └── 盘后 → 返回最近一次缓存（0 API 调用）
```

### 2.6 L5 情报/新闻

| 属性 | 值 |
|------|-----|
| 数据量 | ~10 条/次 |
| 更新频率 | 分钟级 |
| 主存储 | 内存 |
| 内存缓存 TTL | 600s（10 分钟） |
| 更新策略 | 按需拉取 |
| API 调用 | ~6 次/小时 |

## 3. 启动策略

```
首次启动（stock_master 为空）:
  ├── 同步下载 A 股主表（~35s，仅一次）
  └── 不做任何预热

日常启动:
  ├── 检查 SQLite 中行情缓存时间
  │   ├── < 5 分钟 → 跳过预热
  │   └── ≥ 5 分钟 → 后台预热（最多 5 只）
  └── 不拉 K 线、不拉财报、不拉新闻
```

### 预热策略

```
预热条件（AND 关系）:
  ├── 距上次预热 ≥ 4 小时
  ├── SQLite 中最新行情 ≥ 5 分钟
  └── 只预热 5 只股票

预热方式:
  ├── 单只超时 5s
  ├── 间隔 3s（akshare 反爬保护）
  └── 后台线程静默执行
```

## 4. API 调用量估算

```
场景: 单用户，每天使用 2 小时，查看 10 只股票

                当前      优化后     降幅
  行情调用      980 次    245 次     ↓ 75%
  K线调用       100 次     10 次     ↓ 90%
  市场/板块      24 次     24 次      -
  新闻/情报      12 次     12 次      -
  ────────────────────────────────────
  日均合计      ~1100 次  ~290 次    ↓ 74%
```

## 5. 实施优先级

| 优先级 | 改动 | 收益 | 说明 |
|--------|------|------|------|
| 🔴 P0 | 行情盘后不刷新 | 最高 | 周末/夜间 0 调用 |
| 🔴 P0 | K线 SQLite 缓存优先级 | 高 | 同一股票永不重复拉 |
| 🟡 P1 | 预热时间门控 | 中 | 减少无意义预热 |
| 🟡 P1 | 财经数据 SQLite 持久化 | 中 | 7天不重复 |
| 🟢 P2 | 股票主表周期导入 | 低 | 首启后几乎无影响 |
